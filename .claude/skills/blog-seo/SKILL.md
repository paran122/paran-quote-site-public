# SEO 가이드 블로그 스킬

## 기존 blog-post 스킬과의 차이

| | blog-post (기존) | blog-seo (이 스킬) |
|---|---|---|
| 소스 | 우리 행사 사진/후기 | 외부 블로그 리서치 데이터 |
| 톤 | 사례 후기, 현장 중심 | 전문 가이드, 정보 제공 |
| 이미지 | 실제 행사 사진 | AI 생성 (blog-image 스킬 연계) |
| 글자수 | 1,500~2,500자 | 3,000~5,000자 |

## 트리거

이 스킬은 **"SEO 블로그"** 또는 **"리서치"** 키워드가 포함된 경우에 트리거된다.

| 사용자 표현 | 동작 |
|------------|------|
| "SEO 블로그 리서치해줘" + 주제 | 검색 → 리서치 수집 → DB 저장 |
| "SEO 블로그 작성해줘" | 리서치 DB 기반 → 글 작성 → blog_posts 저장 |
| "이번 주 블로그 2개 작성해줘" | 준비된 리서치 중 선택 → 글 작성 |
| "블로그 리서치 현황 알려줘" | blog_research 테이블 통계 조회 |
| "블로그 발행 스케줄 잡아줘" | 카테고리 분산 + 날짜 설정 |

키워드 없이 "블로그 써줘"만 쓴 경우 → "행사 사례 블로그인가요, SEO 가이드 블로그인가요?" 확인

## Part 1: 리서치 수집

### 수집 절차

1. **주제 확인** — 사용자가 지정한 주제 또는 25개 주제 목록에서 선택
2. **검색** — WebSearch로 구글/네이버 검색 (주제당 3~5개 검색어 변형)
3. **본문 수집** — WebFetch로 상위 글 10~20개 본문 읽기
4. **중복 체크** — source_url UNIQUE 제약 + 벡터 유사도 90% 이상 경고
5. **핵심 추출** — 각 글에서 핵심 포인트 3~5개 요약
6. **키워드 추출** — 글에서 SEO 키워드 추출
7. **DB 저장** — blog_research 테이블에 INSERT

### 품질 관리 (수집 시 필터 없음, 사용 시 필터링)

**수집 원칙: 필터 없이 전부 저장**
- 검색 결과에서 본문 접근 가능한 글은 전부 수집
- URL 중복만 차단 (source_url UNIQUE)
- 수집 후 quality_score로 품질 등급 부여

**quality_score 기준:**
| 점수 | 등급 | 기준 |
|------|------|------|
| 0 | 미평가 | 수집 직후 기본값 |
| 1 | 낮음 | 광고성, 500자 미만, 핵심 포인트 1개 이하 |
| 2 | 보통 | 핵심 포인트 2~3개, 일반적 내용 |
| 3 | 높음 | 핵심 포인트 4개+, 구체적 수치/절차/체크리스트 포함 |

**사용 시 필터링:**
```sql
-- 높은 품질만 사용
SELECT * FROM blog_research WHERE topic = '...' AND quality_score >= 2;
-- 전체 확인
SELECT * FROM blog_research WHERE topic = '...' ORDER BY quality_score DESC;
```

### 중복 방지 규칙

```sql
-- 1차: 같은 URL 차단 (UNIQUE 제약)
-- 2차: 유사 내용 감지 (벡터 유사도)
SELECT id, topic, source_title,
  1 - (embedding <=> $1) as similarity
FROM public.blog_research
WHERE 1 - (embedding <=> $1) > 0.9
ORDER BY similarity DESC
LIMIT 5;
```

유사도 90% 이상이면 → "이미 비슷한 내용이 있습니다" 경고 → 저장 여부 사용자 확인

### 카테고리 분류 (blog_posts.category 값)

리서치 테이블(blog_research)의 카테고리와 블로그 글의 카테고리는 다르다.
글 발행 시 아래 3개 중 하나를 사용한다.

| blog_posts 카테고리 | 포함 주제 | 성격 |
|-------------------|----------|------|
| **기획 가이드** | 행사 기획 가이드, 대행사 선정, 기획 vs 대행, 기획서 작성, 견적서 해설, 워크숍 기획, 전시 부스 기획, 교육 행사 기획 | 입문자용 기획 정보 |
| **현장 노하우** | 큐시트, 체크리스트, 등록 데스크, MC 섭외, 만족도 조사, 포스터, 자료집, 기념품, 결과보고서, 입찰 제안서 | 실무자용 팁/템플릿 |
| **행사 정보** | 세미나/컨퍼런스/포럼 차이, 기업 행사 종류, 온라인/하이브리드 행사, 행사장 추천, 대관 비용, 음향 장비, 현수막, 사진/영상 활용 | 유형 비교, 장소, 장비, 비용 정보 |

**기존 행사 사례 블로그 카테고리** (SEO 글과 별도):
- `행사 기획` — 기획 과정/전략 중심 사례
- `행사 후기` — 현장 운영/결과 중심 후기

### 블로그 목록 탭 구조

`[전체]` `[행사 기획]` `[행사 후기]` `[기획 가이드]` `[현장 노하우]` `[행사 정보]`

- 전체 탭: 히어로는 행사 기획/후기만, SEO 글은 하단 EVENT GUIDE 섹션
- 개별 탭: 해당 카테고리만 표시

### 저작권 보호 규칙

- **원문 문장 복사 절대 금지** — 핵심 포인트만 요약하여 저장
- key_points에 저장하는 내용은 원문의 30% 미만으로 압축
- 출처 URL은 반드시 저장 (source_url)
- 수집 데이터는 비공개 (홈페이지에 노출 안 됨)

## Part 2: 글 작성

### 작성 전 확인 (필수)

1. **기존 글 체크** — 같은 target_keyword로 blog_posts에 글이 있는지 확인
2. 있으면 → "기존 글을 업데이트할까요, 새 관련 주제로 작성할까요?" 확인
3. 없으면 → 새 글 작성 진행

### 글 구조 규칙

```
H1: 제목 (타겟 키워드 포함, 1개만)

[도입부] 독자의 문제 정의 (100~200자)
  → "행사를 처음 맡았는데 뭐부터 해야 할지 모르겠다면..."

H2: 섹션 1 (타겟 키워드 또는 관련 키워드 포함)
  본문 + 이미지

H2: 섹션 2
  본문 + 이미지

H2: 섹션 3~7 (총 3~7개)
  본문 + 이미지

[마무리] CTA — 자연스럽게 한 줄
  → "전문 대행이 필요하시다면 파란컴퍼니에 문의해 주세요." + 링크
  → 전화번호/카카오톡 나열 금지 (영업 느낌 방지)
  → 관련 글 2~3개 (아래 내부 링크 규칙 참고)
```

### 글자수 기준

| 항목 | 기준 |
|------|------|
| 최소 | **3,000자** (HTML 태그 제외, 순수 텍스트) |
| 권장 | **4,000~5,000자** |
| H2 섹션당 | 최소 200자 이상 |
| 도입부 | 100~200자 |
| 마무리 CTA | 50~100자 |

3,000자 미만이면 경고하고 보강 권고

### SEO 키워드 배치 규칙

| 위치 | 규칙 |
|------|------|
| 제목 (H1) | 타겟 키워드를 앞부분에 배치 |
| 첫 문단 | 타겟 키워드 1회 자연스럽게 포함 |
| H2 소제목 | 3~7개 중 2개 이상에 관련 키워드 포함 |
| 본문 | 타겟 키워드 자연스럽게 3~5회 (과도한 반복 금지) |
| seo_title | 타겟 키워드 + 핵심 정보 (60자 이내) |
| seo_description | 타겟 키워드 포함 + 행동 유도 (155자 이내) |
| 이미지 alt | 타겟 키워드 또는 관련 키워드 포함 |

### 제목 다양성 규칙 (필수)

**"행사 OOO" 패턴 반복 금지** — 제목이 모두 같은 단어로 시작하면 검색 결과에서 중복처럼 보이고 클릭률이 떨어진다.

| 규칙 | 예시 |
|------|------|
| 같은 첫 단어로 시작하는 제목이 2개 이상 연속 금지 | ❌ "행사 기념품...", "행사 자료집...", "행사 포스터..." |
| 구어체/질문형/숫자형 등 다양한 패턴 혼용 | ✅ "참석자가 진짜 쓰는 기념품 고르는 법", "D-30부터 당일까지: 빠짐없는 준비 체크리스트" |
| 작성 전 기존 글 제목 전체 조회 후 패턴 겹침 확인 | `SELECT title FROM blog_posts ORDER BY published_at DESC;` |

**제목 패턴 종류 (돌려가며 사용):**
- 숫자형: "5가지 핵심 단계", "A to Z"
- 비교형: "직접 기획 vs 대행 맡기기"
- 질문형: "마이크가 안 되면 끝?"
- How-to: "빨리 끝내는 법", "고르는 법"
- 콜론형: "D-30부터 당일까지: 준비 체크리스트"

### 톤앤매너

- **전문 가이드 톤** — "~하세요", "~합니다" (존댓말)
- 객관적이고 정보 중심 (우리 홍보가 아닌 독자 도움 목적)
- 마무리 CTA에서만 자연스럽게 파란컴퍼니 언급
- 네이버 블로그와 차별화: 같은 주제라도 구성/문장 70% 이상 다르게

### 본문 표(table) 사용 주의

HTML 표(`<table>`)는 본문과 글씨체, 행간, 분위기가 달라져서 세련되지 않게 보일 수 있다.

- **표 최소화** — 꼭 필요한 비교 데이터만 표로, 나머지는 리스트(`<ul>/<ol>`)로 대체
- **표 사용 시** — 본문과 동일한 폰트/행간이 적용되도록 별도 스타일링 확인
- **대안 활용** — 비교 정보는 "A는 ~이고, B는 ~입니다" 식의 서술형도 고려
- 표가 2개 이상 연속되면 글의 흐름이 끊기므로 사이에 본문 단락 삽입

### 구체적 비용/수치 작성 주의

- **검증되지 않은 비용은 넣지 않기** — 잘못된 가격 정보는 신뢰도 하락 + 클레임 위험
- 비용 대신 **비중(%)이나 범위**로 표현 (예: "대관비가 전체의 20~30%")
- 구체적 금액을 넣을 경우 → 출처와 시점 명시 (예: "2025년 숨고 평균 기준")
- **파란컴퍼니 자체 데이터가 아닌 외부 수치는 "~부터", "약 ~" 등 유보적 표현** 사용

### 내부 링크 규칙

- 글당 **2~3개** 내부 링크 삽입
- **우선순위**: 같은 SEO 카테고리(기획 가이드/현장 노하우/행사 정보) 블로그 글 → `/blog/{slug}`
- **가이드 글이 부족할 때**: 관련 포트폴리오 → `/work/{slug}`
- 행사 사례 블로그 글도 가능 → `/blog/{slug}`
- /guide 페이지 링크는 지양 (블로그 내 순환 유도)
- 링크 텍스트에 키워드 포함 (예: "행사 견적 가이드 보기")

## Part 3: 발행 규칙

### 예약 발행

- `is_published: true` + `published_at: 미래 날짜` → 자동 예약
- 기존 queries.ts에 `.lte("published_at", new Date().toISOString())` 이미 적용됨

### 카테고리 분산 규칙

같은 카테고리 연속 발행 금지. 발행 전 최근 2개 글의 카테고리를 확인:

```sql
SELECT category FROM public.blog_posts
WHERE is_published = true
ORDER BY published_at DESC LIMIT 2;
```

최근 2개와 다른 카테고리에서 선택

### 발행 빈도

- 기본: 주 2회 (화, 금)
- 사용자가 다른 스케줄을 지정하면 그에 따름

## Part 4: 업데이트 규칙

### 1글 1주제 원칙

- 같은 타겟 키워드로 새 글을 만들지 않음
- 기존 글이 있으면 내용을 업데이트 (slug/URL 유지)

### 키워드 카니발리제이션 체크

글 작성 전 반드시 확인:

```sql
SELECT id, title, slug, seo_title
FROM public.blog_posts
WHERE seo_title ILIKE '%{타겟 키워드}%'
   OR title ILIKE '%{타겟 키워드}%';
```

결과가 있으면 → 경고 + "기존 글 업데이트 vs 다른 키워드로 작성" 선택

### 업데이트 시 처리

1. 본문 내용 갱신
2. 글 상단에 업데이트 날짜 표시: `<p class="text-xs text-slate-400">2026.12 업데이트</p>`
3. `updated_at` 자동 갱신됨 (DB 트리거)
4. slug(URL)은 절대 변경하지 않음

## Part 5: 이미지 처리

글 작성 시 이미지가 필요한 섹션을 표시하고, **blog-image 스킬을 호출**하여 생성한다.

- 포트폴리오 사진은 사용하지 않음 (홈페이지에 이미 있으므로 중복)
- 포트폴리오와의 연결은 텍스트 내부 링크로만

### AI 이미지 초상권·저작권 규칙 (필수)

| 규칙 | 이유 |
|------|------|
| **인물 얼굴이 식별 가능하게 나오면 안 됨** | AI가 실존 인물과 유사한 얼굴을 생성할 수 있어 초상권 문제 발생 |
| **이름표, 명패 등 실명이 보이면 안 됨** | AI가 임의의 한국 이름을 생성하면 실존 인물과 겹칠 위험 |
| **인물은 뒷모습, 멀리서 찍은 실루엣, 손/물건 클로즈업 위주** | 얼굴 식별 불가능하게 |
| **일반인이 등장하는 장면은 모자이크/블러 처리된 스타일로** | 4/10 현수막 글의 서울커피 이미지처럼 |
| 프롬프트에 반드시 포함: `"no visible faces"`, `"shot from behind"`, `"distant wide shot"` 중 하나 | 생성 단계에서 차단 |

**잘못된 예시:** "Korean business person reviewing documents" → 얼굴이 정면으로 나옴
**올바른 예시:** "Hands reviewing documents on a desk, no faces visible, close-up of hands only"

### 이미지 수량 및 배치 규칙

**수량은 같은 카테고리 기존 글과 동일하게 맞춘다.** 작성 전 아래 쿼리로 확인:
```sql
SELECT title, (length(content) - length(replace(content, '<img ', ''))) / length('<img ') as img_count
FROM blog_posts WHERE category = '{같은 카테고리}' AND thumbnail_url IS NOT NULL LIMIT 3;
```

| 규칙 | 기준 |
|------|------|
| 이미지 수 | **같은 카테고리 기존 글과 동일** (현재 기준: 썸네일 1 + 본문 4 = **5장**) |
| 첫 이미지 위치 | 첫 H2 섹션 바로 앞 |
| 이미지 간격 | H2 섹션마다 1장씩 앞에 배치 |
| 썸네일 | 본문 첫 번째 이미지와 동일하게 설정 |
