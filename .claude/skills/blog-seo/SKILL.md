# SEO 가이드 블로그 스킬

## 기존 blog-post 스킬과의 차이

| | blog-post (기존) | blog-seo (이 스킬) |
|---|---|---|
| 소스 | 우리 행사 사진/후기 | 외부 블로그 리서치 데이터 |
| 톤 | 사례 후기, 현장 중심 | 전문 가이드, 정보 제공 |
| 이미지 | 실제 행사 사진 | AI 생성 (blog-image 스킬 연계) |
| 글자수 | 1,500~2,500자 | 3,000~5,000자 |

## 트리거

이 스킬은 **"SEO 블로그"** 또는 **"리서치"** 키워드가 포함된 경우에 트리거된다.

| 사용자 표현 | 동작 |
|------------|------|
| "SEO 블로그 리서치해줘" + 주제 | 검색 → 리서치 수집 → DB 저장 |
| "SEO 블로그 작성해줘" | 리서치 DB 기반 → 글 작성 → blog_posts 저장 |
| "이번 주 블로그 2개 작성해줘" | 준비된 리서치 중 선택 → 글 작성 |
| "블로그 리서치 현황 알려줘" | blog_research 테이블 통계 조회 |
| "블로그 발행 스케줄 잡아줘" | 카테고리 분산 + 날짜 설정 |

키워드 없이 "블로그 써줘"만 쓴 경우 → "행사 사례 블로그인가요, SEO 가이드 블로그인가요?" 확인

## Part 1: 리서치 수집

### 수집 절차

1. **주제 확인** — 사용자가 지정한 주제 또는 25개 주제 목록에서 선택
2. **검색** — WebSearch로 구글/네이버 검색 (주제당 3~5개 검색어 변형)
3. **본문 수집** — WebFetch로 상위 글 10~20개 본문 읽기
4. **중복 체크** — source_url UNIQUE 제약 + 벡터 유사도 90% 이상 경고
5. **핵심 추출** — 각 글에서 핵심 포인트 3~5개 요약
6. **키워드 추출** — 글에서 SEO 키워드 추출
7. **DB 저장** — blog_research 테이블에 INSERT

### 품질 관리 (수집 시 필터 없음, 사용 시 필터링)

**수집 원칙: 필터 없이 전부 저장**
- 검색 결과에서 본문 접근 가능한 글은 전부 수집
- URL 중복만 차단 (source_url UNIQUE)
- 수집 후 quality_score로 품질 등급 부여

**quality_score 기준:**
| 점수 | 등급 | 기준 |
|------|------|------|
| 0 | 미평가 | 수집 직후 기본값 |
| 1 | 낮음 | 광고성, 500자 미만, 핵심 포인트 1개 이하 |
| 2 | 보통 | 핵심 포인트 2~3개, 일반적 내용 |
| 3 | 높음 | 핵심 포인트 4개+, 구체적 수치/절차/체크리스트 포함 |

**사용 시 필터링:**
```sql
-- 높은 품질만 사용
SELECT * FROM blog_research WHERE topic = '...' AND quality_score >= 2;
-- 전체 확인
SELECT * FROM blog_research WHERE topic = '...' ORDER BY quality_score DESC;
```

### 중복 방지 규칙

```sql
-- 1차: 같은 URL 차단 (UNIQUE 제약)
-- 2차: 유사 내용 감지 (벡터 유사도)
SELECT id, topic, source_title,
  1 - (embedding <=> $1) as similarity
FROM public.blog_research
WHERE 1 - (embedding <=> $1) > 0.9
ORDER BY similarity DESC
LIMIT 5;
```

유사도 90% 이상이면 → "이미 비슷한 내용이 있습니다" 경고 → 저장 여부 사용자 확인

### 카테고리 분류 (blog_posts.category 값)

리서치 테이블(blog_research)의 카테고리와 블로그 글의 카테고리는 다르다.
글 발행 시 아래 4개 중 하나를 사용한다.

| blog_posts 카테고리 | 포함 주제 | 성격 |
|-------------------|----------|------|
| **기획 가이드** | 행사 기획 방법론, 대행사 선정, 기획 vs 대행, 기획서 작성, 견적서 해설 | 입문자용 기획 방법론 |
| **현장 노하우** | 큐시트, 체크리스트, 등록 데스크, MC 섭외, 만족도 조사, 포스터, 자료집, 기념품, 결과보고서, 입찰 제안서 | 실무자용 팁/템플릿 |
| **행사 정보** | 세미나/컨퍼런스/포럼 차이, 기업 행사 종류, 온라인/하이브리드 행사, 행사장 추천, 대관 비용, 음향 장비, 현수막, 사진/영상 활용 | 유형 비교, 장소, 장비, 비용 정보 |
| **행사 유형 가이드** | 송년회, 시상식, 창립기념행사, 준공식/기공식, 워크숍, 체육대회, 비전선포식, 취임식, 개업식, 브랜드 런칭 등 | 행사 종류별 A to Z 가이드 |

**기존 행사 사례 블로그 카테고리** (SEO 글과 별도):
- `행사 기획` — 기획 과정/전략 중심 사례
- `행사 후기` — 현장 운영/결과 중심 후기

### 블로그 목록 탭 구조

`[전체]` `[행사 기획]` `[행사 후기]` `[기획 가이드]` `[현장 노하우]` `[행사 정보]` `[행사 유형 가이드]`

- 전체 탭: 히어로는 행사 기획/후기만, SEO 글은 하단 EVENT GUIDE 섹션
- 개별 탭: 해당 카테고리만 표시

### 저작권 보호 규칙

- **원문 문장 복사 절대 금지** — 핵심 포인트만 요약하여 저장
- key_points에 저장하는 내용은 원문의 30% 미만으로 압축
- 출처 URL은 반드시 저장 (source_url)
- 수집 데이터는 비공개 (홈페이지에 노출 안 됨)

## Part 2: 글 작성

### 글 작성 → 검증 → 이미지 생성 프로세스 (필수 순서)

**이 순서를 반드시 지킨다. 단계를 건너뛰거나 순서를 바꾸지 않는다.**

```
1단계: 글 작성 (본문 텍스트만, 이미지 마커 포함)
  ↓
2단계: 글자수 확인 — 기준 미달 시 섹션 추가하여 보강
  ↓
3단계: Gemini 교차검증 (Part 2.5) — 팩트체크 통과 후에만 다음 단계
  ↓
4단계: 이미지 생성 (blog-image 스킬) — 글 내용에 맞는 이미지 생성 + 업로드
  ↓
5단계: DB 등록 — 글 + 이미지 함께 INSERT
```

**주의사항:**
- 글자수 미달인 채로 Gemini 검증을 하지 않는다 (내용 추가 후 다시 검증해야 하므로 이중 작업)
- Gemini 검증 전에 이미지를 생성하지 않는다 (글 내용이 수정되면 이미지 프롬프트도 달라짐)
- 글 작성과 이미지 생성을 분리하지 않는다 — 한 프로세스로 진행

### 작성 전 확인 (필수)

1. **기존 글 체크** — 같은 target_keyword로 blog_posts에 글이 있는지 확인
2. 있으면 → "기존 글을 업데이트할까요, 새 관련 주제로 작성할까요?" 확인
3. 없으면 → 새 글 작성 진행

### 글 구조 규칙

```
H1: 제목 (타겟 키워드 포함, 1개만)

[도입부] 독자의 문제 정의 (100~200자)
  → "행사를 처음 맡았는데 뭐부터 해야 할지 모르겠다면..."

H2: 섹션 1 (타겟 키워드 또는 관련 키워드 포함)
  본문 + 이미지

H2: 섹션 2
  본문 + 이미지

H2: 섹션 3~7 (총 3~7개)
  본문 + 이미지

[FAQ] 자주 묻는 질문 2~3개 (필수)
  → <p><strong>Q. 질문</strong></p><p>A. 답변</p> 형식 필수 (JSON-LD 자동 파싱)
  → FAQ 없는 글은 FAQPage 구조화 데이터가 생성되지 않아 AEO 효과 없음
  → 글 주제에서 실제로 자주 검색되는 질문을 선정

[마무리] CTA + 내부 링크 (필수)
  → "전문 대행이 필요하시다면 파란컴퍼니에 문의해 주세요." + 링크
  → 전화번호/카카오톡 나열 금지 (영업 느낌 방지)
  → 관련 글 내부 링크 **2~3개 필수** (0~1개는 규칙 위반)
```

### 글 작성 완료 후 필수 검증 (생략 불가)

글을 DB에 INSERT하기 전에 아래 항목을 반드시 확인한다.

| 항목 | 기준 | 미달 시 |
|------|------|--------|
| 글자수 | 실무 가이드 2,500자+, 종합 가이드 3,500자+ | 섹션 추가하여 보강 |
| FAQ | 2~3개, Q./A. 패턴 정확히 | FAQ 없으면 JSON-LD 미생성 → AEO 효과 없음 |
| 내부 링크 | 2개 이상 | 기존 글 slug 조회하여 관련 글 연결 |
| 이미지 | 최소 3장 (본문 내용 맞춤 프롬프트) | 이미지 생성 후 업로드 |
| 썸네일 | thumbnail_url 설정 | 첫 번째 이미지와 동일 |
| seo_title | 60자 이내, | 파란컴퍼니 포함 |
| seo_description | 155자 이내, 타겟 키워드 포함 |
| 카테고리 분산 | 최근 2개와 다른 카테고리 | SQL로 확인 필수 |

### AEO (Answer Engine Optimization) 규칙

AI 검색 엔진(ChatGPT, Gemini, Perplexity 등)이 우리 글을 인용하도록 최적화한다.
SEO가 구글 검색 순위라면, AEO는 AI가 답변할 때 출처로 사용되는 것을 목표로 한다.

**1. H2 제목을 질문형으로 작성**

AI는 사용자 질문에 매칭되는 소제목을 찾아 답변한다. H2를 질문형으로 쓰면 매칭 확률이 높아진다.

```
나쁜 예: <h2>워크숍 개요</h2>
좋은 예: <h2>워크숍과 세미나, 뭐가 다를까?</h2>
```

모든 H2를 질문형으로 할 필요는 없고, 핵심 섹션 1~2개를 질문형으로 작성한다.

**2. H2 바로 아래에 핵심 답변 1~2문장**

AI는 소제목 바로 다음 문장을 답변으로 추출한다. 첫 문장에 핵심 결론을 넣는다.

```
<h2>워크숍과 세미나, 뭐가 다를까?</h2>
<p>세미나는 전문가가 발표하고 참가자가 듣는 행사이고, 워크숍은 참가자 전원이 직접 참여해서 결과물을 만드는 행사입니다.</p>
← AI가 이 문장을 답변으로 추출
```

**3. 구조화된 데이터 적극 사용**

AI는 리스트(`<ul>/<ol>`)와 핵심 용어 볼드(`<strong>`)를 잘 파싱한다.

- 3개 이상 나열할 때 → 반드시 리스트 사용
- 핵심 개념/용어 → `<strong>`으로 강조
- 비교 정보 → 리스트 내 "A — 설명. B — 설명." 패턴

**4. 명확한 정의를 글 초반에 배치**

글의 첫 문단에 핵심 개념의 정의를 넣는다. AI가 "~란 무엇인가?" 질문에 사용한다.

```
좋은 예: "워크숍은 참가자가 직접 체험하며 결과물을 만드는 참여형 행사입니다."
```

**5. 숫자/단계/체크리스트 구조**

"N단계", "N가지", "체크리스트" 형태의 구조는 AI가 답변으로 인용하기 가장 좋은 포맷이다.

```
좋은 예: "워크숍 기획 5단계: 1단계 목표 정의 → 2단계 참가자 구성 → ..."
```

**6. FAQ 패턴 (글 하단) — JSON-LD 자동 생성됨**

글 하단에 자주 묻는 질문 2~3개를 Q&A 형태로 추가하면 AI 인용 확률이 높아진다.
**중요: `blog/[slug]/page.tsx`에서 아래 HTML 패턴을 자동 파싱하여 FAQPage JSON-LD를 생성한다.**
반드시 아래 형식을 정확히 지켜야 FAQ 구조화 데이터가 구글/AI에 노출된다.

```html
<h2>자주 묻는 질문</h2>
<p><strong>Q. 워크숍 적정 인원은?</strong></p>
<p>A. 한 테이블 5~6명, 전체 20~30명이 일반적으로 운영하기 좋은 규모입니다.</p>
```

**FAQ HTML 형식 규칙 (JSON-LD 파싱용):**
- 질문: `<p><strong>Q. 질문 내용</strong></p>` — 반드시 `<strong>` 안에 `Q.`으로 시작
- 답변: `<p>A. 답변 내용</p>` — 반드시 `A.`으로 시작
- 질문과 답변은 연속된 `<p>` 태그여야 함 (사이에 다른 태그 삽입 금지)
- 글당 FAQ 2~3개 권장 (5개 이상은 과도)

**적용 체크리스트:**
- [ ] H2 중 1~2개가 질문형인가?
- [ ] 각 H2 바로 아래에 핵심 답변 1~2문장이 있는가?
- [ ] 리스트/볼드로 구조화되어 있는가?
- [ ] 글 초반에 핵심 개념 정의가 있는가?
- [ ] 숫자/단계 구조가 사용되었는가?

### 글자수 기준

**고정 최소치 없음. 검색 의도에 맞는 적정 길이로 작성한다.**

| 글 유형 | 적정 길이 | 예시 |
|---------|----------|------|
| 간단한 비교/정의 | 1,500~2,500자 | "세미나 vs 컨퍼런스 차이" |
| 실무 가이드/절차 | 2,500~3,500자 | "행사 기획서 작성법", "큐시트 작성법" |
| 종합 가이드/비교 | 3,500~5,000자 | "행사장 추천", "행사 기획 완벽 가이드" |

| 공통 기준 | |
|----------|---|
| H2 섹션당 | 최소 200자 이상 |
| 도입부 | 100~200자 |
| 마무리 CTA | 50~100자 |

**판단 기준:** 같은 키워드로 Google 상위 3개 글의 평균 길이를 참고.
경쟁 글보다 짧으면 불리하고, 의미 없이 채워 늘리면 이탈률 증가.

### SEO 키워드 배치 규칙

| 위치 | 규칙 |
|------|------|
| 제목 (H1) | 타겟 키워드를 앞부분에 배치 |
| 첫 문단 | 타겟 키워드 1회 자연스럽게 포함 |
| H2 소제목 | 3~7개 중 2개 이상에 관련 키워드 포함 |
| 본문 | 타겟 키워드 자연스럽게 3~5회 (과도한 반복 금지) |
| seo_title | title-guide.md 참조 |
| seo_description | title-guide.md 참조 |
| 이미지 alt | 타겟 키워드 또는 관련 키워드 포함 |

### seo_title · seo_description 동기화

**상세 규칙은 `title-guide.md` 참조.** title 변경 시 seo_title, seo_description도 반드시 함께 변경한다.

### 톤앤매너

- **전문 가이드 톤** — "~하세요", "~합니다" (존댓말)
- 객관적이고 정보 중심 (우리 홍보가 아닌 독자 도움 목적)
- 마무리 CTA에서만 자연스럽게 파란컴퍼니 언급
- 네이버 블로그와 차별화: 같은 주제라도 구성/문장 70% 이상 다르게

### 본문 표(table) 사용 주의

HTML 표(`<table>`)는 본문과 글씨체, 행간, 분위기가 달라져서 세련되지 않게 보일 수 있다.

- **표 최소화** — 꼭 필요한 비교 데이터만 표로, 나머지는 리스트(`<ul>/<ol>`)로 대체
- **표 사용 시** — 본문과 동일한 폰트/행간이 적용되도록 별도 스타일링 확인
- **대안 활용** — 비교 정보는 "A는 ~이고, B는 ~입니다" 식의 서술형도 고려
- 표가 2개 이상 연속되면 글의 흐름이 끊기므로 사이에 본문 단락 삽입

### 구체적 비용/수치 작성 주의

- **검증되지 않은 비용은 넣지 않기** — 잘못된 가격 정보는 신뢰도 하락 + 클레임 위험
- 비용 대신 **비중(%)이나 범위**로 표현 (예: "대관비가 전체의 20~30%")
- 구체적 금액을 넣을 경우 → 출처와 시점 명시 (예: "2025년 숨고 평균 기준")
- **파란컴퍼니 자체 데이터가 아닌 외부 수치는 "~부터", "약 ~" 등 유보적 표현** 사용

### 내부 링크 규칙

- 글당 **2~3개** 내부 링크 삽입
- **우선순위**: 같은 SEO 카테고리(기획 가이드/현장 노하우/행사 정보) 블로그 글 → `/blog/{slug}`
- **가이드 글이 부족할 때**: 관련 포트폴리오 → `/work/{slug}`
- 행사 사례 블로그 글도 가능 → `/blog/{slug}`
- /guide 페이지 링크는 지양 (블로그 내 순환 유도)
- 링크 텍스트에 키워드 포함 (예: "행사 견적 가이드 보기")

## Part 3: 발행 규칙

### 예약 발행

- `is_published: true` + `published_at: 미래 날짜` → 자동 예약

SELECT category FROM public.blog_posts
WHERE is_published = true

최근 2개와 다른 카테고리에서 선택

## Part 4: 업데이트 규칙

### 1글 1주제 원칙

- 같은 타겟 키워드로 새 글을 만들지 않음
- 기존 글이 있으면 내용을 업데이트 (slug/URL 유지)

### 키워드 카니발리제이션 체크

글 작성 전 반드시 확인:

```sql
SELECT id, title, slug, seo_title
FROM public.blog_posts
WHERE seo_title ILIKE '%{타겟 키워드}%'
   OR title ILIKE '%{타겟 키워드}%';
```

결과가 있으면 → 경고 + "기존 글 업데이트 vs 다른 키워드로 작성" 선택

1. 본문 내용 갱신
2. 글 상단에 업데이트 날짜 표시: `<p class="text-xs text-slate-400">2026.12 업데이트</p>`
3. `updated_at` 자동 갱신됨 (DB 트리거)
4. slug(URL)은 절대 변경하지 않음

## Part 5: 이미지 처리

글 작성 시 이미지가 필요한 섹션을 표시하고, **blog-image 스킬을 호출**하여 생성한다.

- 포트폴리오 사진은 사용하지 않음 (홈페이지에 이미 있으므로 중복)
- 포트폴리오와의 연결은 텍스트 내부 링크로만

### AI 이미지 초상권·저작권 규칙 (필수)

| 규칙 | 이유 |
|------|------|
| **인물 얼굴이 식별 가능하게 나오면 안 됨** | AI가 실존 인물과 유사한 얼굴을 생성할 수 있어 초상권 문제 발생 |
| **이름표, 명패 등 실명이 보이면 안 됨** | AI가 임의의 한국 이름을 생성하면 실존 인물과 겹칠 위험 |
| **인물은 뒷모습, 멀리서 찍은 실루엣, 손/물건 클로즈업 위주** | 얼굴 식별 불가능하게 |
| **일반인이 등장하는 장면은 모자이크/블러 처리된 스타일로** | 4/10 현수막 글의 서울커피 이미지처럼 |
| 프롬프트에 반드시 포함: `"no visible faces"`, `"shot from behind"`, `"distant wide shot"` 중 하나 | 생성 단계에서 차단 |

**잘못된 예시:** "Korean business person reviewing documents" → 얼굴이 정면으로 나옴
**올바른 예시:** "Hands reviewing documents on a desk, no faces visible, close-up of hands only"

### AI 이미지 텍스트/한글 규칙 (필수)

AI 이미지에 한글 텍스트가 포함되면 높은 확률로 깨진 글자가 나온다.

| 규칙 | 이유 |
|------|------|
| **프롬프트에 한글 텍스트 요청 금지** | AI가 한글을 정확히 렌더링하지 못함 (예: "겸토", "진헁즁" 등 깨진 글자 생성) |
| **화이트보드, 포스트잇, 간판 등 텍스트가 보이는 요소 최소화** | 텍스트가 포함된 오브젝트는 깨진 글자가 나올 위험 |
| 프롬프트에 `"no text"`, `"no visible text"`, `"no writing"` 중 하나 포함 | 생성 단계에서 텍스트 렌더링 차단 |
| 텍스트가 필요한 이미지는 생성 후 편집 도구로 한글 오버레이 | AI 생성 시 텍스트 품질 보장 불가 |

### 같은 글 내 이미지 분위기 통일 (필수)

한 블로그 글에 포함되는 모든 이미지는 **동일한 분위기와 톤**을 유지해야 한다.

| 규칙 | 구체적 기준 |
|------|-----------|
| **배경 통일** | 한국 배경이면 전부 한국, 실내면 전부 실내. 한 장은 한국풍 + 한 장은 외국풍 금지 |
| **색감/조명 통일** | 첫 번째 이미지의 색감(따뜻한/차가운, 밝은/어두운)을 기준으로 나머지도 맞춤 |
| **촬영 스타일 통일** | 포토리얼리스틱이면 전부 포토리얼리스틱, 일러스트면 전부 일러스트 |
| **프롬프트에 기준 이미지 참조** | 두 번째 이미지부터는 첫 번째 이미지의 스타일 키워드를 프롬프트에 반복 포함 |

**구현 방법:** 첫 번째 이미지 프롬프트에서 배경/스타일 키워드를 정하고, 나머지 이미지 프롬프트에 동일 키워드를 복사한다.
```
예시:
이미지 1: "... in a modern Korean office, warm lighting, professional photo style"
이미지 2: "... in a modern Korean office, warm lighting, professional photo style"  ← 동일
이미지 3: "... in a modern Korean office, warm lighting, professional photo style"  ← 동일
```

### 이미지 수량 및 배치 규칙

**글 내용에 맞게 적절한 수만큼 생성하되, SEO를 위해 최소 3장 이상 포함한다.**

| 규칙 | 기준 |
|------|------|
| 이미지 수 | **최소 3장 ~ 최대 5장** (썸네일 1 + 본문 2~4장) |
| 첫 이미지 위치 | 첫 H2 섹션 바로 앞 |
| 배치 기준 | H2 섹션 2~3개마다 1장씩, 글 전체에 고르게 분포 |
| 썸네일 | 본문 첫 번째 이미지와 동일하게 설정 |
| alt 텍스트 | 타겟 키워드 포함 (SEO) |
| 색상 | **blog-image 스킬의 색상 규칙 참고** — 특정 팔레트에 국한하지 않고 주제에 맞는 자유로운 색상 사용 |
